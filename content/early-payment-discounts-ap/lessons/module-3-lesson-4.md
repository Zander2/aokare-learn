## Key Metrics

A discount program generates significant data. The challenge is not collecting metrics — it is selecting the ones that drive action.

### The Metrics Hierarchy

**Tier 1: Executive Metrics** (CFO dashboard, monthly review)
- Total discount savings (dollars)
- Program ROI (savings / cost)
- Annualized return on early-deployed cash

**Tier 2: Operational Metrics** (AP manager dashboard, weekly review)
- Discount capture rate (captured / available)
- Outreach volume and response rate
- Average cycle time from identification to payment
- Error rate (missed windows, incorrect amounts)

**Tier 3: Diagnostic Metrics** (agent operators, daily monitoring)
- Response classification accuracy
- Follow-up conversion rates
- Supplier-level acceptance trends
- Cash utilization rate (early payment budget used / allocated)

### Calculating Total Savings

The savings calculation must be precise and auditable:

```python
def calculate_period_savings(start_date, end_date):
    captured_discounts = get_captured_discounts(start_date, end_date)

    total_savings = sum(d.original_amount - d.paid_amount for d in captured_discounts)
    total_invoices_discounted = len(captured_discounts)
    total_original_value = sum(d.original_amount for d in captured_discounts)
    weighted_avg_discount = total_savings / total_original_value if total_original_value else 0

    # Calculate what was available vs. what was captured
    all_eligible = get_eligible_invoices(start_date, end_date)
    available_savings = sum(
        inv.amount * inv.discount_percent / 100
        for inv in all_eligible
    )
    capture_rate = total_savings / available_savings if available_savings else 0

    return {
        "total_savings": total_savings,
        "capture_rate": capture_rate,
        "invoices_discounted": total_invoices_discounted,
        "weighted_avg_discount_pct": weighted_avg_discount * 100,
        "available_savings": available_savings,
        "missed_savings": available_savings - total_savings,
    }
```

**Important:** Only count savings where the discount was actually applied. If the agent scheduled an early payment but the supplier did not apply the discount (it happens), that is not a saving — it is an overpayment. Reconciliation against supplier statements is essential.

### Identifying Underperformance

Compare each segment's actual performance against benchmarks:

```python
def identify_underperforming_segments(performance_data, benchmarks):
    underperformers = []

    for segment, metrics in performance_data.items():
        benchmark = benchmarks[segment]

        if metrics.acceptance_rate < benchmark.acceptance_rate * 0.8:
            underperformers.append({
                "segment": segment,
                "metric": "acceptance_rate",
                "actual": metrics.acceptance_rate,
                "benchmark": benchmark.acceptance_rate,
                "gap": benchmark.acceptance_rate - metrics.acceptance_rate,
                "suggested_action": "Review messaging and discount rates for this segment"
            })

        if metrics.response_time > benchmark.response_time * 1.5:
            underperformers.append({
                "segment": segment,
                "metric": "response_time",
                "actual": metrics.response_time,
                "benchmark": benchmark.response_time,
                "suggested_action": "Check email deliverability and contact freshness"
            })

    return underperformers
```

### Common Causes and Fixes

| Symptom | Likely Cause | Fix |
|---------|-------------|-----|
| Low acceptance rate, specific industry | Discount rate too aggressive for industry norms | Reduce opening offer for this industry |
| Declining acceptance rate over time | Suppliers experiencing "proposal fatigue" | Reduce outreach frequency, improve personalization |
| High counter-offer rate | Opening offer too far from supplier expectations | Anchor closer to historical acceptance points |
| Low response rate | Email deliverability issues or wrong contacts | Verify email addresses, check spam folder deliverability |
| High response rate but low acceptance | Good engagement but poor terms alignment | Adjust terms based on counter-offer patterns |

## A/B Testing

Systematic experimentation improves the agent's performance over time. A/B testing is the mechanism.

### What to Test

| Element | Variable A | Variable B | Success Metric |
|---------|-----------|-----------|----------------|
| Subject line | "Early Payment Opportunity" | "Save $X on Invoice #Y" | Open rate |
| Opening paragraph | Formal tone | Conversational tone | Response rate |
| Discount rate | 2.0% opening | 1.5% opening | Acceptance rate |
| Follow-up timing | 3 days | 5 days | Conversion on follow-up |
| Call to action | "Reply Accepted" | Button link | Acceptance rate |
| Bundle vs. single | One email per invoice | Bundle 3 invoices | Revenue per email |

### Setting Up an A/B Test

```python
def setup_ab_test(test_name, variants, allocation_percent=50, min_sample_size=100):
    """
    Randomly assign suppliers to test variants.
    Ensure each supplier consistently sees the same variant (no flip-flopping).
    """
    test = {
        "name": test_name,
        "variants": variants,
        "allocation": allocation_percent,
        "min_sample": min_sample_size,
        "start_date": today(),
        "status": "running",
        "results": {v: {"sent": 0, "responses": 0, "acceptances": 0} for v in variants}
    }

    return test

def assign_variant(supplier_id, test):
    # Consistent assignment: hash supplier ID to determine variant
    hash_val = hash(f"{supplier_id}:{test['name']}") % 100
    if hash_val < test["allocation"]:
        return test["variants"][0]
    else:
        return test["variants"][1]
```

### Analyzing Results

Use statistical significance testing before declaring a winner:

```python
from scipy import stats

def analyze_ab_test(test):
    variant_a = test["results"][test["variants"][0]]
    variant_b = test["results"][test["variants"][1]]

    # Acceptance rate comparison
    rate_a = variant_a["acceptances"] / variant_a["sent"] if variant_a["sent"] else 0
    rate_b = variant_b["acceptances"] / variant_b["sent"] if variant_b["sent"] else 0

    # Chi-squared test for significance
    observed = [
        [variant_a["acceptances"], variant_a["sent"] - variant_a["acceptances"]],
        [variant_b["acceptances"], variant_b["sent"] - variant_b["acceptances"]]
    ]
    chi2, p_value, _, _ = stats.chi2_contingency(observed)

    return {
        "rate_a": rate_a,
        "rate_b": rate_b,
        "lift": (rate_b - rate_a) / rate_a if rate_a else 0,
        "p_value": p_value,
        "significant": p_value < 0.05,
        "recommendation": "adopt_b" if rate_b > rate_a and p_value < 0.05 else "continue_testing"
    }
```

**Do not** declare results significant until you have at least 100 observations per variant. Early results fluctuate wildly. A test that shows 60% vs. 40% acceptance after 10 proposals per variant may converge to 48% vs. 52% after 200.

### Test Velocity

Run one test at a time per supplier segment to avoid confounding variables. With 40-50 proposals per week, a typical test reaches statistical significance in 4-6 weeks. Plan a continuous testing calendar:

- Month 1-2: Subject line optimization
- Month 3-4: Discount rate optimization by segment
- Month 5-6: Follow-up timing and cadence
- Month 7-8: Message personalization depth
- Ongoing: Re-test previous winners against new challengers

## Quarterly Business Reviews

Every quarter, present program performance to leadership. Structure the review around four questions:

**1. What did we save?**
Total savings, incremental savings over the prior period, and savings as a percentage of eligible AP spend.

**2. What did it cost?**
Platform costs, integration maintenance, FTE time spent on program management and exception handling.

**3. What is working?**
Top-performing supplier segments, successful A/B test results, process improvements implemented.

**4. What needs to change?**
Underperforming segments, missed opportunities, system issues, and proposed adjustments for the next quarter.

Include a forward-looking projection:

```
Q3 Projected Savings:    $340,000 - $380,000
Key Driver:              15 new suppliers onboarding to discount program
Risk Factor:             Q3 cash position expected to be tighter (seasonal)
Proposed Budget Increase: $500K/month → $600K/month early payment budget
Expected Incremental Savings: $45,000/quarter from budget increase
```

This structured reporting builds organizational confidence in the program and justifies continued investment. It also creates accountability — if the agent's performance degrades, the quarterly review surfaces it before the damage compounds.
